# v0.5.0 Milestone Direction

**Theme: Composable Workflow Toolkit, General-Purpose Workspaces, and Safe Real Mutation**
**Status: DRAFT — direction seeded; detailed planning (specs + packets) is a dedicated planning pass after v0.4.0 ships**

---

## 1. Origin

This document carries forward the broad "composable toolkit" deliverables that the
original (pre-reframe) v0.4.0 contract bundled together. When v0.4.0 was narrowed to
the **Proactive Assistance** release (see `v0.4.0_contract.md`), the toolkit/recipe/
apply work — which is genuinely a separate, larger release — moved here.

`grain suggest` and the upstream feedback loop (`grain report` / `grain notes` /
opt-in telemetry) **graduated into v0.4.0** and are no longer v0.5.0 scope. v0.5.0
builds on that shipped `suggest` foundation.

## 2. Candidate Core Deliverables

Carried forward from the original toolkit contract, plus the audit's recommended
general-purpose direction:

1. **General-purpose / non-code workspaces** — `workspace_kind: code | knowledge | mixed`;
   relax git/branch/AST assumptions so Grain runs knowledge/research/ops workflows, not
   just code repos. *(Headline new direction — the audit's #1 recommendation.)*
   - **Canonical workspace marker + manifest.** Promote `grain.toml` to the single
     authoritative workspace marker AND manifest home (carry `workspace_kind`, version/
     branch policy there), and have `grain init` create it. Today the markers are
     split-brain: `grain init` writes `docs/runtime/PROJECT_RULES.md` (the real resolver
     marker) but NOT `grain.toml`, so of the live workspaces only ~6/11 carry a
     `grain.toml` (the `grain` product itself has none), while 17 stray `grain.toml`
     stubs sit in uninitialized dirs. Migration required: backfill the markerless live
     workspaces, resolve/remove the stubs, make the resolver authoritative on
     `grain.toml`. *(Surfaced 2026-06-26 via the grain-fleet aggregator; do NOT flip ad
     hoc — it's a resolver change with a migration. `workspace_model.md` is
     canonical/approval-gated, so this note lives in the working contract until the
     planning pass.)*
2. **`grain recipe` execution** — `recipe.yaml` + `prompt.md` + `task_template.md`;
   `grain recipe list/show/run/scaffold`; bundled starter recipes.
3. **`grain recipe suggest`** — propose which recipes to create/run from workspace
   signals, reusing the v0.4.0 suggest proposal model. *(Founder-requested.)*
4. **External signal ingestion** — a local-first, file-backed signal inbox
   (`docs/working/signals/`) that an external driver (DAEMON/Sovereign) writes into so
   calendar/email/Notion-derived signals influence `grain suggest` without any network
   code inside Grain.
5. **Safe apply graduation** — gated in-place `apply` for `.docx` / spreadsheet /
   Obsidian when validators fully pass + a packet-scoped backup exists.
6. **Grain-as-engine contract** — versioned, expanded MCP surface so DAEMON/Sovereign
   (and future "familiars") can drive the full loop headlessly, not just create tasks.
7. **Toolkit contract + workspace model** — bilateral capability declarations and
   multi-repo workspace resolution.
8. **Dev/runtime alignment diagnostics** — detect drift between the installed CLI and
   the active repo source.
9. **Context token-budget proxy** — make the token-efficiency claim *measurable*, not
   assumed. `grain context build/show` emits an estimated token cost for the assembled
   bundle, ranks the heaviest sources, and surfaces source-trimming hints; optionally a
   per-task/per-phase token-cost column in `grain metrics`. Today Grain is *designed* for
   token efficiency but never reports it — this closes that gap. Matters most for
   familiars: an agent driving Grain unattended needs a visible context budget to stay
   economical. *(Founder-requested 2026-06-25; revives the old contract's shelved
   "Observability & Token-Efficiency" section.)*
10. **Graduated ceremony / right-sized workflow ("quick lane")** — not everything should
    be a full packet, and some work shouldn't be in Grain at all. Make ceremony match task
    weight: build on the existing `task create --simple` and `task close --quick`, add a
    `grain quick "<desc>"` one-shot lane (no context assembly, no review gate; logged to a
    lightweight ledger rather than a 6-file packet dir), a triviality heuristic that
    *recommends* the quick lane (and that `grain suggest` can apply), and explicit,
    guilt-free guidance that trivial/throwaway work can skip Grain entirely. Counters the
    current "one active task packet at a time / never bypass gates" bias that taxes small
    fixes. *(Founder-requested 2026-06-25.)*
11. **Package self-update notice** — tell end users when a newer `grain-kit` is published.
    Today nothing does: `grain upgrade` only syncs workspace FILES to the installed
    version, and `upgrade_policy`/min_version only warns when the installed CLI is older
    than the *workspace* declares — there is NO PyPI release check, so a solo user with no
    min_version set never learns a new release shipped. Add an opt-in, cached (daily, in
    `.grain`), non-blocking PyPI version check that prints a one-line "grain-kit X.Y.Z
    available (you have A.B.C) — run `uv tool upgrade grain-kit`" notice; fail-silent on
    network error; `GRAIN_NO_UPDATE_CHECK=1` escape hatch. Distinct from #8 (installed-vs-
    repo-source drift) — this is installed-vs-latest-released.
    BOTH surfaces required: (a) HUMAN — the one-line CLI notice above; (b) FAMILIAR —
    expose installed-vs-latest version + update-available as structured data through the
    Grain-as-engine contract (#6) / a `grain version --check --format json`, so an agent
    can detect a new release and either self-upgrade through its environment (if the keeper
    permits) or surface it to the keeper. *(Founder-requested 2026-06-25.)*

## 3. Guiding Principle

Every surface should be drivable end-to-end by a **familiar** (agent) with no human and
no browser — API/file-backed/MCP over interactive prompts, structured JSON everywhere.

## 4. Non-Goals (unchanged)

Hosted services or daemons; broad new adapter families beyond office/document/Obsidian;
replacing the CLI as the canonical surface; cloud collaboration.
