# v0.5.0 Milestone Direction

**Theme: Composable Workflow Toolkit, General-Purpose Workspaces, and Safe Real Mutation**
**Status: DRAFT — direction seeded; detailed planning (specs + packets) is a dedicated planning pass after v0.4.0 ships**

---

## 1. Origin

This document carries forward the broad "composable toolkit" deliverables that the
original (pre-reframe) v0.4.0 contract bundled together. When v0.4.0 was narrowed to
the **Proactive Assistance** release (see `v0.4.0_contract.md`), the toolkit/recipe/
apply work — which is genuinely a separate, larger release — moved here.

`grain suggest` and the upstream feedback loop (`grain report` / `grain notes` /
opt-in telemetry) **graduated into v0.4.0** and are no longer v0.5.0 scope. v0.5.0
builds on that shipped `suggest` foundation.

## 2. Candidate Core Deliverables

Carried forward from the original toolkit contract, plus the audit's recommended
general-purpose direction:

1. **General-purpose / non-code workspaces** — `workspace_kind: code | knowledge | mixed`;
   relax git/branch/AST assumptions so Grain runs knowledge/research/ops workflows, not
   just code repos. *(Headline new direction — the audit's #1 recommendation.)*
2. **`grain recipe` execution** — `recipe.yaml` + `prompt.md` + `task_template.md`;
   `grain recipe list/show/run/scaffold`; bundled starter recipes.
3. **`grain recipe suggest`** — propose which recipes to create/run from workspace
   signals, reusing the v0.4.0 suggest proposal model. *(Founder-requested.)*
4. **External signal ingestion** — a local-first, file-backed signal inbox
   (`docs/working/signals/`) that an external driver (DAEMON/Sovereign) writes into so
   calendar/email/Notion-derived signals influence `grain suggest` without any network
   code inside Grain.
5. **Safe apply graduation** — gated in-place `apply` for `.docx` / spreadsheet /
   Obsidian when validators fully pass + a packet-scoped backup exists.
6. **Grain-as-engine contract** — versioned, expanded MCP surface so DAEMON/Sovereign
   (and future "familiars") can drive the full loop headlessly, not just create tasks.
7. **Toolkit contract + workspace model** — bilateral capability declarations and
   multi-repo workspace resolution.
8. **Dev/runtime alignment diagnostics** — detect drift between the installed CLI and
   the active repo source.

## 3. Guiding Principle

Every surface should be drivable end-to-end by a **familiar** (agent) with no human and
no browser — API/file-backed/MCP over interactive prompts, structured JSON everywhere.

## 4. Non-Goals (unchanged)

Hosted services or daemons; broad new adapter families beyond office/document/Obsidian;
replacing the CLI as the canonical surface; cloud collaboration.

## 5. Next Step

Run a v0.5.0 planning phase (lock specs, write task packets) once v0.4.0 is merged and
the `suggest`/feedback foundation is stable. The locked specs already exist for several
of these (`recipe_spec.md`, `toolkit_contract.md`, `workspace_model.md`,
`apply_graduation.md`, `feedback_spec.md`).
