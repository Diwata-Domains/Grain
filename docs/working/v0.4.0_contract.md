# v0.4.0 Milestone Contract

**Theme: Composable Workflow Toolkit and Safe Real Mutation**
**Status: LOCKED**
**Locked: 2026-06-11 — Phase 30-T01 (TASK-0190)**

---

## 1. Theme

v0.4.0 turns Grain from a strong single-project operator surface into a composable workflow toolkit. A toolkit that other projects in the same workspace can wire into, that can execute repeatable workflow patterns across project types, and that enforces packet-first discipline reliably regardless of which agent is driving.

The "safe real mutation" half addresses the gap between v0.3.0's `propose` mode (always safe, never in-place) and what real workflows require: updating documents in place when validation is strong enough, graduating from proposal-only to apply-capable for surfaces where Grain has enough validation coverage to make it trustworthy.

This release deepens what already landed. It does not expand into new agent capabilities, cloud features, or broad new adapter families.

## 2. Core Deliverables

All six must ship for v0.4.0 to close.

### 2.1 — Agent Enforcement Hardening
Enforce packet-first discipline through CLI gates, git hooks, and file-backed state — not through agent-specific prompt instructions. Enforcement must work for any agent (Claude Code, Codex, Gemini, custom tools) and for a human running `git commit` directly.

Deliverables:
- `grain workflow guard` — standalone guard command; callable from hooks, CI, any agent
- `grain hooks install` — writes pre-commit and post-checkout hooks
- `prompts/workflow.resume.md` — agent-agnostic session resume protocol seeded in every workspace
- Hardened `grain workflow next` state machine: `packet_required` stop reason, done-task stale-pointer fix
- Hard rule additions to bundled `PROJECT_RULES.md`
- AGENTS.md block updated to reference `workflow.resume.md` by path

Spec source: P30-T08 (TASK-0197)

### 2.2 — Scaffold Seeding Fixes
`grain init` and `grain onboard` must seed a complete workspace — not a minimal one. Every canonical and working doc type must have a seeded stub so agents don't start sessions with missing reference files.

Deliverables:
- `grain init` seeds: `product_scope.md`, `architecture.md`, `decisions.md`, `landscape.md`, `backlog.md`, `current_focus.md`, `open_questions.md`, `change_proposals.md`, `roadmap.md`, `current_task.md`, `CHANGELOG.md`, `tooling_notes.md`, `docs/working/proposals/` dir
- `grain init --name <name> --type <type>` flags added
- `tooling_notes read_when: never` manifest bug fixed (should be `always`)
- `grain onboard` seeding gaps closed to match `grain init`

Spec source: P30-T06 (TASK-0195)

### 2.3 — `grain suggest`
Proactive task suggestion with human approval gate. Grain surfaces what to do next — existing ready tasks or new task proposals derived from workspace signals — and writes them as file-backed proposals. Nothing is acted on without operator approval.

Deliverables:
- `grain suggest` command: `pick-up` and `new-task` suggestion types
- Proposal files written to `docs/working/proposals/` with `SUG-YYYYMMDD-NNN` IDs
- `grain suggest accept <id>` and `grain suggest dismiss <id>`
- Signal inputs: backlog ready tasks, open OQs, high-severity tooling notes, recent git commits

Spec source: P30-T07 (TASK-0196)

### 2.4 — Toolkit Contract + Workspace Model
Explicit bilateral contracts that let Grain and sibling toolkit apps declare capabilities and requirements without relying on chat-memory glue. Multi-repo workspace resolution so a Grain workspace in a monorepo can link to sibling projects and aggregate context across them.

Deliverables:
- `toolkit_contract.yaml` bilateral declaration format and schema
- `grain context link` — registers a sibling project context
- `grain workspace list` — shows all linked workspaces in the current project tree
- Monorepo resolution order spec

Spec source: P30-T02 (TASK-0191)

### 2.5 — `grain recipe` Execution
Recipe execution surfaces that make repeatable workflow patterns first-class. A recipe is a named, parameterized workflow slice — not just a prompt template, but the full execute → review → close pattern configured for a specific use case.

Deliverables:
- `grain recipe` command group: `list`, `run`, `scaffold`, `show`
- Recipe packet scaffold format
- Recipe prompt template component model
- Initial bundled recipe examples (at least two from v0.3.0 recipe targets)

Spec source: P30-T03 (TASK-0192)

### 2.6 — Apply Graduation Criteria
Spec and first safe `apply` slices for the artifact surfaces where Grain's validation coverage is now strong enough to support in-place mutation. The `propose` → `apply` promotion is conditional on all validators passing; partial validation results always produce `propose`, never `apply`.

Deliverables:
- Per-surface apply graduation criteria (`apply_graduation.md`): `.docx`, spreadsheet, Obsidian
- At least one surface graduates from proposal-only to apply-capable
- Apply safety gate enforced in CLI: 100% required validators pass + at least one non-empty change summary + zero unacknowledged residual risk flags

Spec source: P30-T04 (TASK-0193)

### 2.7 — `grain docs audit`
Proactive consistency check for the entire working document layer, not just the active task packet. Detects stale pointers, backlog/packet mismatches, aging open questions, un-triaged tooling notes, and structural gaps across all registered docs. Read-only; <2s execution; configurable thresholds. Complements `grain workflow reconcile` (packet-focused) with a broader doc-health view.

Spec source: P30-T10 (TASK-0199)

---

### 2.8 — Archiving model
Coherent archiving across three surfaces: automatic (working doc snapshots at phase close; proposal pruning when signals resolve), suggested (milestone-archive and workspace-idle suggestions via `grain suggest`), and explicit (`grain archive snapshot`, `grain archive milestone <version>`). Consistent with existing `grain phase archive` behavior.

Spec source: P30-T11 (TASK-0200)

### 2.9 — CLI output ergonomics
JSON output contract hardening (every automation-relevant command gets a stable `--format json`); canonical stop reason vocabulary; text output style consistency; `grain status` — single command for complete workspace state.

Spec source: P30-T12 (TASK-0201)

### 2.10 — TUI extension for v0.4.0 command surface
Extend the v0.3.0 TUI (Python + Textual) with panels for `grain suggest`, `grain docs audit`, `grain archive`, `grain doctor`, and `grain recipe`. TUI remains a pure presentation layer over existing CLI commands. No new business logic in the TUI layer.

Spec source: P30-T13 (TASK-0202)

### 2.11 — Upgrade enforcement
`upgrade_policy` block in `docs_manifest.yaml` declares a minimum Grain version requirement per workspace. Warn-only mode (default): stderr banner on every invocation. Enforce mode: blocks all commands except `grain upgrade`, `grain --version`, `grain doctor` until user upgrades. `grain upgrade` automatically ratchets `min_version` on success. Grace period configurable. Override logged to tooling_notes.

Spec source: P30-T14 (TASK-0203)

- Broad new adapter families beyond office/document and Obsidian
- Replacing or redesigning the CLI as the canonical surface
- Database adapter expansion (v0.3.0 scope, already landed)
- Homebrew/tap distribution
- TUI redesign or embedded agent terminals

## 5. Open Questions — Resolved

### OQ-1: Unit of reuse in `grain recipe`

**Decision: The unit of reuse is the workflow slice** — a named, parameterized sequence of Grain workflow steps (execute → review → close) with an associated prompt template and packet scaffold template. Individual prompt templates and packet templates are recipe components, not recipes. A recipe describes a complete use-case-level pattern.

Rationale: Every v0.3.0 recipe target ("update PRD from source inputs", "revise meeting notes", "fix Obsidian vault links") is a multi-step pattern that already fits execute/review/close. The recipe layer adds parameterization and a named entrypoint — it configures the workflow engine, it doesn't replace it. A recipe that is only a prompt template (no workflow slice) is not a recipe; it belongs in `prompts/`.

### OQ-2: Serialization format for toolkit contract

**Decision: YAML** for authoring and storage. JSON for machine output (`--format json`).

Rationale: YAML is already Grain's manifest format (`docs_manifest.yaml`, `workflow_loop.yaml`). The toolkit contract is a human-editable spec artifact — the same authoring model as other Grain manifests. No new format is introduced. The contract YAML follows the same structural conventions as existing Grain manifest files.

### OQ-3: Validation threshold for `propose` → `apply`

**Decision: All required validator classes must return zero failures. At least one validator must emit a non-empty change summary. Residual risk notes must be empty or explicitly operator-acknowledged before apply proceeds.**

No partial pass qualifies for `apply`. The single exception: validators marked `advisory_only` in the adapter profile downgrade their failures to warnings only — they do not block apply.

Rationale: `apply` is an in-place mutation. The bar must be strict enough that operators trust it without checking every detail. A partial validation model would erode trust faster than it would help adoption. When in doubt, Grain falls back to `propose`.

### OQ-4: Signals and quality bar for `grain suggest`

**Decision — Input signals (priority order):**
1. `docs/working/backlog.md` — tasks with `ready` status in active or next phase
2. `docs/working/open_questions.md` — questions with `decision_needed` or `blocking` status
3. `docs/working/tooling_notes.md` — `open` entries with `high` severity referencing Grain commands
4. Recent git history (last 3 commits) — used to avoid suggesting work just completed and to infer natural next steps

**Decision — Minimum quality bar:**

Pick-up type:
- Task must be `ready` status
- Must be in the active phase or the next blocked phase
- Must not be the currently `in_progress` task
- Must not be a task touched by any of the last 3 commits

New-task type:
- Must cite a traceable input signal (specific OQ ID, tooling_notes row, or git commit pattern)
- Must state a concrete objective naming a specific deliverable (not "improve X")
- Must not duplicate an existing task title at ≥70% token similarity

Any suggestion that cannot cite a traceable signal is filtered before it reaches the user. The signal reference is included in the proposal file.

| Phase | Theme | Core Deliverables | Source |
|-------|-------|-------------------|--------|

| **Phase 31** | DX Hardening — bugs + enforcement + scaffold + audit + archive + CLI ergonomics + upgrade gate | DX-1–DX-5 + 2.1 + 2.2 + 2.7 + 2.8 + 2.9 + 2.11 | T06, T08, T10–T12, T14, DX items |

**Why Phase 31 first:** Enforcement and scaffold fixes are prerequisites for every phase that follows. A `grain suggest` that fires without an open packet is as broken as an unenforced `grain workflow next`. DX bugs must be fixed before building on top of them.

**Why `grain suggest` before the toolkit contract:** `grain suggest` is user-facing and doesn't depend on the toolkit contract shape being final. Getting suggestion behavior in front of operators early gives feedback before the cross-tool integration work begins.

**Phase 36 and 37 are conditional:** They ship if Phases 31–35 complete on schedule without consuming the full planning-derived budget. If they slip to v0.5.0, the core release is still complete.
