## 1. Theme

v0.4.0 makes Grain proactive. Instead of only answering "what is the next legal step?",
Grain now surfaces *what to work on* when there is no obvious next task, turns local
workflow friction into queryable and reportable work, measures how the workflow actually
performs, and lays the opt-in telemetry contract the rest of the Diwata stack will consume.

It deepens the shipped workflow core. It does not add cloud features, daemons, or new
adapter families. Everything stays file-backed and local-first; every external action is
opt-in and transparent.

## 3. Two GitHub feedback surfaces (2.8) — why both

- **`grain report`** (canonical, URL-based): upstream path — a Grain user sends friction to
  the maintainer repo. Browser-confirmed, no token, privacy-preserving (no file contents/paths/PII).
- **`grain notes publish`** (API-based): downstream path — files a note as an issue in the
  *user's own* repo via `GRAIN_GITHUB_TOKEN`. No browser, no human click — the only path that
  works when a **familiar** (agent) drives Grain headlessly. Turns the friction log into tracked work.

## 4. Principles

- File-backed, local-first; no background services or hidden state.
- All suggestion/orchestration output is a proposal — nothing acted on without explicit accept.
- Telemetry and GitHub actions are opt-in and transparent; nothing leaves the machine by default.
- Every automation-relevant command keeps a stable `--format json`.

## 5. Non-Goals

Hosted services/daemons; telemetry transport/aggregation (Pulse owns that — Grain only emits);
broad new adapter families; replacing the CLI as the canonical surface; recipe/toolkit/apply
(those are v0.5.0 — see `v0.5.0_contract.md`).
